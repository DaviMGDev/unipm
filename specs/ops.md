### Shell completion setup

```bash
# Bash
eval "$(unipm completion bash)"

# Zsh
eval "$(unipm completion zsh)"

# Fish
unipm completion fish | source
```

Users add the appropriate line to their shell config file (`.bashrc`, `.zshrc`, `config.fish`). The completion script is generated by Cobra's built-in completion engine and requires no external dependencies.

### Verification after install

```bash

```

## Configuration & state

### File locations

| File | Path | Purpose | Created by |
|------|------|---------|------------|
| Config | `~/.unipm/config.yaml` | Distrobox definitions, cache TTL, search timeout | First run (auto-generated with defaults) |
| State | `~/.unipm/state.json` | Tracks packages installed via unipm | `unipm install` (append), `unipm uninstall` (remove) |
| Cache | `~/.unipm/cache.json` | Package name cache for tab-completion | `unipm search` (populate), TTL-based expiry |

All files live under `~/.unipm/`. The directory is created on first run if it doesn't exist.

### Config defaults

```yaml
distrobox: {}       # No containers configured by default
cache_ttl: 86400    # 24 hours
search_timeout: 10  # seconds
```

### State file versioning

The state file includes a `"version"` field. If unipm encounters an unknown version, it errors with a message asking the user to upgrade or manually migrate. This prevents silent corruption from format changes across releases.

## Privilege escalation

unipm itself NEVER runs as root. When a backend requires elevated privileges (e.g., `apt install`, `pacman -S`), the adapter shells out to `sudo <command>`. Only that subprocess runs elevated. The user's `sudo` configuration (password requirement, `NOPASSWD` rules) determines the interactive experience.

Backends that do not require root (npm global, pip --user, brew, flatpak user mode, appimage) run without privilege escalation.

## Non-functional requirements

### Performance

| Operation | Target | Notes |
|-----------|--------|-------|
| `unipm search <query>` | Results start appearing < 2 s; full table < 10 s | Parallel goroutines; timeout per adapter |
| `unipm install <pkg>` | Delegated to native PM; no additional overhead beyond the subprocess | unipm's own overhead is sub-100 ms |
| `unipm sources` | < 100 ms | Only checks `$PATH` and container existence â€” no network |
| Tab-completion | < 50 ms for cached results | Local cache read; no network |

### Reliability

| Metric | Target | Notes |
|--------|--------|-------|
| State file corruption | Never â€” atomic writes (write to temp + rename) | Prevents partial writes on crash or power loss |
| Adapter timeout handling | Graceful degradation â€” partial results with warning | Failed adapter does not block other adapters |
| Error messages | Every error path returns a human-readable message + non-zero exit code | No silent failures |

## Security

### Threat model

**Assets**: User's package installation state (`state.json`) and configuration (`config.yaml`).

**Adversaries**:

| Adversary | Motivation | Capability |
|-----------|------------|------------|
| Malicious package (supply chain) | Install malware via a compromised upstream package | Medium â€” depends on upstream registry security; unipm delegates to native PMs which have their own verification (GPG, checksums) |
| Local privilege escalation | Exploit sudo delegation to gain root | Low â€” unipm does not parse or interpret sudo output; subprocess arguments are constructed with known-safe strings |

**Attack surfaces**: Minimal â€” unipm has no network listeners, no daemon mode, no IPC, and no plugin system that loads external code.

### Controls

| Control | Implementation |
|---------|---------------|
| No SUID/setuid | unipm binary is never installed with elevated permissions |
| Scoped sudo | Only backend subprocesses run with `sudo`; unipm itself runs as the invoking user |
| Config/state permissions | `~/.unipm/` directory and files created with `0700` / `0600` permissions |
| Atomic state writes | State file written to temp file + renamed to prevent corruption |
| Dependency scanning | `govulncheck` in CI; gitleaks for secret detection |

### Vulnerability management

1. **Detection**: `govulncheck` on every PR; dependabot/renovate for Go module updates.
2. **Triage**: Critical/High CVEs in Go stdlib or key dependencies (Cobra, Bubbletea) prioritized.
3. **SLAs**: Critical 48h / High 7d / Medium next release.
